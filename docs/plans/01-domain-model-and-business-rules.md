# Plan 01: Domain Model And Business Rules

## Goal

Define the canonical backend domain model and business rules for families, members, CANs, compliance statuses, computed tasks, dashboard metrics, and reports.

## Context

The generated HTML currently uses two in-memory arrays: `families` and `members`. Each member has exactly one CAN number and belongs to one family. Dashboard cards, family summaries, task lists, and reports are computed directly from member status fields.

This plan converts that browser-only model into backend rules that every later API, import, report, and frontend integration must follow.

## In Scope

- Family, member, CAN, KYC, PayEezz, contact, nominee, bank, and audit-facing concepts.
- Required enum values and derived status meanings.
- Dashboard formulas and family completion formulas.
- Computed task generation rules.
- Report row definitions.
- Validation rules that are independent of framework or database implementation.

## Out Of Scope

- Physical database migrations.
- FastAPI route implementation.
- Authentication and role enforcement.
- PII encryption implementation.
- MFU file parsing.

## Design

### Entities

`Family` is the top-level customer grouping.

Required family fields:

- `id`: UUID generated by backend.
- `family_code`: unique human-readable code, for example `FAM-001`.
- `family_head_name`: display name of family head.
- `primary_rm_id`: user id for the assigned RM.
- `remarks`: optional free text.
- `created_at`, `updated_at`, `deleted_at`.

Derived family fields:

- `total_members`: count of active members.
- `total_cans`: count of active members because v1 uses one CAN per member.
- `last_updated_at`: max of family `updated_at` and active member `updated_at`.
- `kyc_completion_pct`
- `mobile_verification_pct`
- `email_verification_pct`
- `nominee_verification_pct`
- `payeezz_completion_pct`

`Member` represents one family member and one CAN.

Required member fields:

- `id`: UUID generated by backend.
- `family_id`: owning family id.
- `name`: full member name.
- `can_number`: required unique CAN string.
- `kyc_status`: one of the KYC enum values.
- `mobile_status`: one of the verification enum values.
- `email_status`: one of the verification enum values.
- `nominee_status`: one of the verification enum values.
- `payeezz_status`: one of the PayEezz enum values.
- `created_at`, `updated_at`, `deleted_at`.

Optional member fields:

- `pan`
- `date_of_birth`
- `mobile`
- `email`
- `bank_name`
- `bank_account_number`
- `ifsc_code`
- `payeezz_amount`
- `payeezz_start_date`
- `remarks`

### Status Enums

KYC status:

- `Validated`
- `Registered`
- `No KYC`

Verification status:

- `Verified`
- `Not Verified`

PayEezz status:

- `Not Available`
- `Sent for Approval`
- `Aggregator Accepted`

### Compliance Rules

KYC:

- `Validated` means compliant.
- `Registered` means re-KYC pending.
- `No KYC` means KYC pending.
- KYC pending count = members where status is `Registered` or `No KYC`.
- KYC completion percentage = `Validated / total_members * 100`.

PayEezz:

- `Aggregator Accepted` means completed.
- `Sent for Approval` means in progress.
- `Not Available` means pending initiation.
- PayEezz pending count = `Not Available` plus `Sent for Approval`.
- PayEezz completion percentage = `Aggregator Accepted / total_members * 100`.

Contact:

- Mobile completion percentage = members where `mobile_status = Verified`.
- Email completion percentage = members where `email_status = Verified`.
- Contact pending filter = members where mobile or email is `Not Verified`.

Nominee:

- Nominee completion percentage = members where `nominee_status = Verified`.
- Nominee pending filter = members where nominee is `Not Verified`.

Percentage behavior:

- If denominator is zero, return `0`.
- Use integer rounding with standard rounding to nearest whole number.
- Percent values in API responses should include both `count` and `percentage`.

### Computed Task Rules

Tasks are not stored in v1. They are generated from current member status fields.

Generate a `kyc` task:

- If `kyc_status = No KYC`: priority `high`, label `No KYC`, description `KYC not done`.
- If `kyc_status = Registered`: priority `medium`, label `Re-KYC`, description `Re-KYC pending`.

Generate a `payeezz` task:

- If `payeezz_status = Not Available`: priority `high`, label `Not Setup`, description `PayEezz mandate not initiated`.
- If `payeezz_status = Sent for Approval`: priority `medium`, label `Pending`, description `PayEezz sent, awaiting acceptance`.

Generate verification tasks:

- If `mobile_status = Not Verified`: type `mobile`, priority `medium`, label `Unverified`, description `Mobile number not verified in MFU`.
- If `email_status = Not Verified`: type `email`, priority `low`, label `Unverified`, description `Email address not verified in MFU`.
- If `nominee_status = Not Verified`: type `nominee`, priority `medium`, label `Not Verified`, description `Nominee details not verified`.

Task response fields:

- `type`
- `priority`
- `member_id`
- `member_name`
- `family_id`
- `family_head_name`
- `family_code`
- `rm_id`
- `rm_name`
- `can_number_masked`: CAN display value for task lists, using only the last 6 characters.
- `description`
- `label`

### Report Definitions

`kyc_pending`:

- Include active members where `kyc_status != Validated`.
- Columns: name, CAN, PAN masked, KYC status, family head, family code, RM, last updated.

`payeezz_pending`:

- Include active members where `payeezz_status != Aggregator Accepted`.
- Columns: name, CAN, PayEezz status, bank name, account masked, family head, family code, RM.

`contact_pending`:

- Include active members where mobile, email, or nominee status is `Not Verified`.
- Columns: name, CAN, mobile status, email status, nominee status, family head, family code, RM.

`family_compliance`:

- Include active families.
- Columns: family head, family code, RM, members, KYC percentage, PayEezz percentage, mobile percentage, email percentage, nominee percentage.

`rm_tasks`:

- Include computed pending tasks grouped by RM.
- Columns: RM, KYC count, PayEezz count, mobile count, email count, nominee count, total.

`full`:

- Include all active members.
- Columns: name, CAN, PAN masked, DOB, KYC, mobile status, email status, nominee status, PayEezz, bank name, IFSC, family head, family code, RM, last updated.

## Implementation Steps

1. Define enum constants in a shared domain module before creating database models.
2. Implement pure functions for completion percentages and task generation.
3. Implement report definition objects that name the filter, columns, and sort order for each report type.
4. Add unit tests for every enum, formula, and task generation rule.
5. Use the same functions from dashboard, reports, and import commit flows to avoid duplicated logic.

## Subagent Usage

- Use `cavecrew-investigator` to inspect current frontend JavaScript calculations before implementing formula functions.
- Use the main thread for final rule definitions because they affect schemas, APIs, imports, reports, and tests.
- Use `cavecrew-reviewer` after implementation to verify that all formulas match this document.
- Do not use `cavecrew-builder` for this plan unless a later change is limited to one known formula file.

## Test Plan

- Unit tests for `pct(0, 0)`, partial completion, and full completion.
- Unit tests for KYC pending, PayEezz pending, contact pending, and nominee pending filters.
- Unit tests for each generated task type and priority.
- Unit tests proving no task is generated for fully compliant member states.
- Unit tests for report filters using a mixed fixture with all status combinations.

## Acceptance Criteria

- The enum labels exactly match the master plan.
- Dashboard, task, and report calculations use shared domain functions.
- Every business rule in this plan has at least one test.
- Zero-member families return zero percentages without errors.

## Risks & Mitigations

- Risk: formulas drift between dashboard and reports. Mitigation: centralize formula functions.
- Risk: future MFU API uses different status labels. Mitigation: add an adapter mapping layer and keep internal enums stable.
- Risk: one-CAN-per-member becomes insufficient. Mitigation: isolate member/CAN assumptions in this plan and avoid naming tables as generic clients where possible.

## Dependencies

None. This plan is the foundation for all later plans.
